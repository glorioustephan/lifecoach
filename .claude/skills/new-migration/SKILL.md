# New Migration — /new-migration

Scaffold migration: **$1**

Naming convention: `<unix-ts>_<verb>_<noun>.sql`
Example input `add_tax_lots` → `1716912345_add_tax_lots.sql`

## Step 1 — Compute timestamp

Run:

```bash
date +%s
```

Capture the output as `<ts>`. This is the migration's unix timestamp prefix.

## Step 2 — Validate short-name format

`$1` must match `[a-z][a-z0-9_]*` (snake_case, no hyphens, starts with letter).
If it does not, stop:

```
Error: short-name "$1" is invalid. Use snake_case (e.g. add_tax_lots, alter_embeddings_dim).
The name should read as <verb>_<noun> per the migration naming convention.
```

## Step 3 — Check for collisions

Glob `packages/core/src/memory/migrations/*_$1.sql`. If any file matches, stop:

```
Error: a migration with short-name "$1" already exists. Inspect existing file(s):
<list matches>
```

## Step 4 — Write migration stub

Write `packages/core/src/memory/migrations/<ts>_$1.sql`:

```sql
-- =============================================================================
-- Migration: <ts>_$1.sql
-- Purpose:   FILL IN — one-line description of what this migration does
-- Created:   <today ISO date>
-- Linked spec: FILL IN — e.g. docs/specs/plugins/<name>.md (or "none")
-- Author:    <invoking agent name>
-- =============================================================================
--
-- Instructions:
--   1. Replace "FILL IN" fields above.
--   2. Write idempotent SQL (use IF NOT EXISTS / IF EXISTS guards).
--   3. Never DROP columns or tables without a paired rollback section below.
--   4. Dim invariant: any vector column must use dim = VOYAGE_EMBED_DIM (1024).
--   5. Run: sqlite3 <db-path> < this_file  to test locally before committing.
--
-- Rollback (optional — delete if not applicable):
--   Describe the inverse operation or paste the rollback SQL here.
-- =============================================================================

-- TODO: write migration SQL below this line


```

## Step 5 — Append to migrations reference

Read `docs/reference/migrations.md`. If it does not exist, create it with this header:

```markdown
<!-- GENERATED by new-migration skill. Do not edit the table rows by hand.
     Re-run /new-migration to add entries; memory-systems-engineer maintains this file. -->

# Migrations Reference

| Timestamp | Short name | Status | Linked spec | Notes |
|-----------|------------|--------|-------------|-------|
```

Append a new table row:

```
| <ts> | $1 | pending | FILL IN | |
```

(Engineer updates Status from `pending` → `applied` after running the migration.)

## Step 6 — Print confirmation

```
Created migration: packages/core/src/memory/migrations/<ts>_$1.sql
Updated index:     docs/reference/migrations.md

Next steps:
  1. Open the stub and write the migration SQL.
  2. Fill in "Purpose" and "Linked spec" in the header block.
  3. Test locally: sqlite3 <db-path> < packages/core/src/memory/migrations/<ts>_$1.sql
  4. Update docs/reference/migrations.md status to "applied" after running.
  5. Run pnpm -r typecheck to confirm no breakage.
```
