# Docs Rules

Rules for all content under `docs/`. Applies to human authors, domain advisors, and the `technical-writer` agent.

## Dual-audience writing

Every document serves one or more audiences: `agents` (AI subagents consuming structured context) and `humans` (developers reading for understanding).

- **Lead with what humans need** — orientation paragraph, plain-language summary, decision rationale.
- **Append agent-action sections** — structured data, frontmatter, checklists, exact field names — so agents can parse them reliably.
- Never write a document that is machine-readable only (no prose) or human-readable only (no structured metadata).

## Frontmatter schema

Every `.md` file under `docs/` must open with YAML frontmatter. Required fields:

```yaml
---
title: <Human-readable title>
slug: <kebab-slug matching filename, no extension>
audience: [agents, humans]          # one or both values
owner: <agent-name>                 # e.g. technical-writer, financial-advisor
status: draft|ready|in-progress|done|superseded
created: <YYYY-MM-DD>
updated: <YYYY-MM-DD>               # must be updated whenever the file changes
source: authored|generated
generator: <skill-or-script>        # required only when source: generated
code_paths:                         # optional; powers post-edit-domain-sync hook
  - packages/core/src/memory/**
consumers:                          # optional; powers /handoff command
  - memory-systems-engineer
produces:                           # optional; what this brief/spec creates
  - schema: tax_lots
pair:                               # optional; brief↔spec link
  - docs/specs/plugins/<name>.md
last_implemented: null              # date string when implementing engineer ships it
---
```

Optional fields may be omitted, but if the file is a brief targeting engineers, `consumers` and `code_paths` are strongly recommended to enable automated routing.

## Generated vs. authored separation

Files with `source: generated` must carry a top-of-body banner immediately after the closing `---` of frontmatter:

```
<!-- GENERATED by <generator>. Do not edit by hand. Re-run via /<skill-name>. -->
```

Files with `source: authored` must NOT carry this banner.

The `technical-writer` agent owns regeneration of `docs/reference/**`. Never hand-edit a generated file — fix the generator instead.

## @path link convention

To inline content from another file into a document (for agent consumption), use the `@path` syntax:

```markdown
@packages/schemas/src/index.ts
```

This causes Claude Code to inline the referenced file's content at session load. Use sparingly — only for small, high-signal files that agents need verbatim. Do not `@path` large source files.

## Freshness expectation

The `updated` frontmatter field must be within **30 days** of the latest `git log -1 --format=%ai` date for any file listed in `code_paths`.

The `doc-refresh` skill audits drift. If a doc is stale:
1. The `technical-writer` agent is responsible for regenerated docs.
2. The owning domain advisor is responsible for brief freshness.
3. The implementing engineer is responsible for bumping `last_implemented` when they ship.

## Section ordering for briefs

Briefs under `docs/briefs/` must follow this section order (per domain-advisors rule):
Problem → Recommendation → Data shapes → Prompts/Rubrics → Evaluation criteria → Risks → Handoff checklist.

## Section ordering for specs

Specs under `docs/specs/` must follow: Overview → Input contract → Output contract → Error cases → Examples → Changelog.

## Status lifecycle

Valid `status` transitions:

```
draft → ready → in-progress → done
             ↘ needs-input ↗
        done → superseded
```

Do not skip states. A brief marked `ready` with no `consumers` list is valid but will not be picked up automatically by technical agents.
