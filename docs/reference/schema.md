# Database Schema Reference

SQLite schema for lifecoach. All timestamps are Unix milliseconds unless noted. Managed via sequential migration files in `packages/core/src/storage/migrations/`.

---

## Core tables

### `profile`
Key/value store for user profile data (JSON-encoded values).

### `sessions` / `messages`
Conversation session and message history.

### `facts`
Semantic memory facts (user profile data, preferences, observations). Soft-deleted via `valid_to`.

### `documents`
Ingested documents (markdown, PDFs, CSVs). Referenced by measurements and embeddings.

### `measurements`
Time-series numeric observations (weight, HRV, net_worth, monthly_burn, savings_rate, etc.). `metric` + `recorded_at` are the natural key.

### `reflections`
Structured reflections generated by the Reflector.

### `insights`
Inbox-style insights surfaced to the user.

### `embedding_refs`
Maps embedding rowids (in the `embeddings` vec0 virtual table) to the entity they describe. Key columns: `ref_type`, `ref_id`, `chunk_index`, `text`, `model`, `dimension`, `text_hash`, `embedded_at`, `source_updated_at`.

`ref_type` values: `fact`, `document`, `message`, `reflection`, `task`, `finance`, `goal`, `milestone`.

### `embeddings` (virtual, sqlite-vec)
`vec0` virtual table storing `FLOAT[N]` embedding vectors. Dimension N = `VOYAGE_EMBED_DIM` env var (default 1024). Created at runtime by `ensureEmbeddingTable()` — not in SQL migration files.

### `meta`
Lightweight key/value store for schema metadata (embedding_dim, schema_version, etc.).

## Financial tables (migration `008_financial.sql`)

### `accounts`
Monarch-synced bank/investment accounts. `type` ∈ `checking|savings|credit_card|investment|debt|other`.

### `transactions`
Monarch-synced transactions. Key fields:
- `category_group_type` (added migration `1779971644`) — value `transfer` means the transaction is a cash movement between owned accounts and MUST be excluded from income/expense rollups.
- `is_transfer` (added migration `1779979153`) — Monarch's own boolean flag (`1`/`0`/`NULL`). `NULL` means unknown (rows synced before migration). When `1`, this is the priority-1 (tier-0) signal in `isTransferTxn()`, overriding all category-based heuristics.

### `holdings`
Point-in-time investment portfolio snapshots.

### `budgets`
Monthly budget limits by category.

### `financial_insights`
Claude-synthesized financial insights with `source_data_ids` (JSON array of contributing transaction IDs).

### `transaction_overrides` (migration `014_categorization_overrides.sql`)
User corrections to Monarch's categorization. Applied at read time — re-syncs never overwrite them.

### `categorization_rules` (migration `014_categorization_overrides.sql`)
Pattern-based merchant categorization rules. Applied after per-transaction overrides.

## Goal / task tables

### `tasks`, `goals`, `projects`, `milestones`
Task and goal tracking. Goals include WOOP fields (added migration `1779943871`). Tasks have a goal link (migration `1779943873`).

### `goal_signals`, `goal_evidence` (migration `1779945576`)
Evidence-backed signals for goal progress.

## Transfer-exclusion note

Transfer exclusion uses a four-tier priority cascade in `isTransferTxn()` (`packages/core/src/financial/transfer.ts`):

1. **Tier 0** — `is_transfer = 1` (Monarch's own boolean, migration `1779979153`). Short-circuits all heuristics.
2. **Tier 1** — `category_group_type = 'transfer'` (migration `1779971644`).
3. **Tier 2** — `category` name contains substrings: `transfer`, `credit card payment`, `loan payment`, `balance adjustment` (fallback for older rows).
4. **Tier 3** — account-pair cross-reference (caller-supplied set of internal account IDs).

All four conditions independently produce exclusion; earlier tiers short-circuit later ones.

## Financial rollup computation (no new table)

G5 itemization is computed on demand — no new table was added. The canonical rollup lives in `packages/core/src/financial/rollup.ts` and returns a `MonthlyRollup` struct with:

- `contributingTxIds: string[]` — IDs of all non-transfer transactions that contributed to income/expense totals.
- `transferTxIds: string[]` — IDs of excluded transfer transactions (audit trail).
- `guardsPassed: GuardResult[]` — G1–G6 guard evaluation results.
- `outlierMonthDetected: boolean` — true if any month in the window exceeds 1.5× the 6-month median.

This is computed from live transaction rows each time it is called; no rollup rows are persisted. The `measurements` table continues to store daily scalar snapshots (`monthly_burn_mtd`, `savings_rate_mtd`, etc.) for trend tracking, but these do not carry lineage. For any cited figure, call `financial_monthly_rollup` to get the full itemized rollup, then `get_rollup_contributors` to drill down to the raw rows.
