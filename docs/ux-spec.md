# Lifecoach Web UI — UX & Information Architecture Spec

> Phase 2 of the roadmap: replace the CLI with a browser surface accessible from iPhone (primary) and laptop (secondary) over Tailscale. Single user (James). Visual mood: calm, trusted, personal — not enterprise.

This document specifies **structure, behavior, and flow only**. Visual design (color, type, spacing tokens) belongs to a separate visual spec. Component-level styling is owned by the prompt-kit + shadcn/ui + Tailwind v4 stack.

---

## 0. First principles

These show up repeatedly below; stating them once.

1. **Chat is the home.** Every other view is a side trip. The user should never be more than one tap from the composer.
2. **Mobile is the dominant device.** Anything that doesn't work one-handed on an iPhone is broken. Laptop is an enhancement, not a separate design.
3. **The agent has memory; the UI should not pretend it doesn't.** Surface what the agent knows about you, what it just did, what changed.
4. **Privacy mood.** Single user, allow-listed, no analytics, no upsell. The UI should feel like a personal notebook, not a product.
5. **Asynchronous-friendly.** Insights, syncs, ingestions happen in the background. The UI must surface results without demanding attention.
6. **No multi-anything.** No teams, no orgs, no sharing, no public surfaces, no "members." Anywhere a generic SaaS would have a person-picker, we have nothing.

## 1. View inventory

Six top-level views. Each named in the URL and addressable by deep link.

| # | View          | Route                    | One-line purpose                                                    |
|---|---------------|--------------------------|---------------------------------------------------------------------|
| 1 | **Chat**      | `/`  (and `/c/$id`)      | The agent. Where 90% of time is spent. Streaming, tool-aware.       |
| 2 | **Inbox**     | `/inbox`                 | Today's agent-generated insights + anything that needs your eyes.   |
| 3 | **Memory**    | `/memory`                | Browse + search what the agent knows: facts, documents, measurements, reflections. |
| 4 | **Tasks**     | `/tasks`                 | Your Todoist, lightly. Lifecoach view on top, not a Todoist clone.  |
| 5 | **Sources**   | `/sources`               | Integration status + ingestion history. Where data comes from.      |
| 6 | **Settings**  | `/settings`              | Profile (identity facts), preferences, account, dangerous actions.  |

**What lives outside this list (intentionally):**

- A separate "history" or "conversations" view → folded into Chat (as a sheet/drawer of past sessions).
- A separate "documents" view → folded into Memory.
- A separate "measurements" or "labs" view → folded into Memory with a tab.
- A separate "calendar" or "email" view → those are inputs to the agent, not destinations. Calendar context belongs in Chat ("what's on today?") and Inbox.

### View depth detail

#### 1. Chat (`/` and `/c/$sessionId`)
**Purpose:** Talk to the coach. Stream responses. Render tool calls. Recall context.

**Default:** opens to a fresh session OR resumes the most recent session if it's <24h old (configurable in Settings).

**Anatomy:**
- prompt-kit `ChatContainer` filling the main area
- A message list of streaming messages with tool-call disclosure (see §4.1)
- Composer pinned to bottom (prompt-kit `PromptInput` with model picker hidden — we use one model)
- Slash-command palette inline in the composer (`/ingest`, `/forget`, `/status`, `/recall`, `/reflect`)
- Header bar: session title (auto-generated by agent or "Today, 3:14pm"), context-sensitive overflow menu
- Empty state for brand-new sessions (see §8)

**Sub-states:** new-session, resumed-session, viewing-historical-session (read-only banner appears).

#### 2. Inbox (`/inbox`)
**Purpose:** Surface what the agent has *proactively* produced: insights, suggested Todoist changes, anomalies in measurements ("HRV down 12% over 7 days"), gentle nudges. This is the view James opens first thing in the morning.

**Anatomy:**
- Stack of "insight cards" sorted by importance × recency
- Each card: topic, body, rationale (disclosable), source-fact chips
- Per-card actions: **Discuss in chat** (opens new session pre-seeded with the insight), **Mark acted-on**, **Dismiss**, **Snooze**
- Filter chips at top: Today / This week / All
- Section: "Reflections" — daily/weekly LLM-generated summaries, collapsed by default
- Empty state: "Nothing to surface this morning. Your last reflection was [date]."

#### 3. Memory (`/memory`)
**Purpose:** Browse what the agent knows. Trust through visibility. Also: edit/forget.

**Anatomy:** Sub-tabs along the top.
- **Facts** — searchable list grouped by category (health, preference, recipe, routine, goal, relationship, other). Each row: subject, body, source, confidence pill, valid-from/to. Tap → fact detail sheet with edit/forget actions.
- **Documents** — ingested files, sorted by ingested-at. Title, source, mime, size, chunk count. Tap → document detail sheet (preview, re-ingest, forget).
- **Measurements** — metric picker + time-series sparkline. Default view shows pinned metrics (weight, HRV, fasting glucose, etc. — user-configurable). Tap a metric → larger chart with raw data table.
- **Reflections** — chronological list of daily/weekly summaries. Tap → full reflection.
- **Search** (global across all four) — accessible via search icon in the tab bar; semantic recall using the same backend `recall` tool.

#### 4. Tasks (`/tasks`)
**Purpose:** A coach-aware view of Todoist. Not a Todoist replacement — Todoist's own app is better for raw task management. This view is for: triage-with-the-agent, see-what-the-agent-changed, manual re-sync.

**Anatomy:**
- Top: "Today" section (due-today + overdue, grouped by project, sorted by priority)
- "This week" section (due in next 7 days)
- "Inbox" section (no project)
- "Recent agent actions" pinned at top when present: a strip showing the last N tasks the agent added/completed/rescheduled, with **Undo** and **Confirm** affordances
- Tap a task → sheet: title, project, labels, priority, due, "Discuss in chat" action, "Open in Todoist" link
- FAB / header action: **Sync now** (manual sync trigger; see §4.4)

#### 5. Sources (`/sources`)
**Purpose:** "Where does my data come from, and is it healthy?" Connection status + ingestion log + add-source.

**Anatomy:**
- Cards for each connector: Todoist, Google Calendar, Gmail, Capacities, Apple Health, File-drop. Each shows: connected/disconnected, last sync (relative time), next sync (if scheduled), error state with disclosure
- Per-card actions: **Sync now**, **Disconnect**, **View activity** (opens activity sheet)
- "File drop" card always present (not really an OAuth-based source): shows the watched folder path, last N ingested files, drag-drop / upload affordance
- "Recent ingestion activity" feed at bottom: chronological list of every parse → embed → extract event; expand for chunk count, extracted facts/measurements, errors

#### 6. Settings (`/settings`)
**Purpose:** Profile (identity facts the agent always knows about you), preferences, account, danger zone.

**Anatomy:** Vertical sections, each collapsible.
- **Profile** — well-known identity keys (name, preferredName, DOB, dosha, blood type, allergies, dietary preferences, location, timezone, goals) as a structured form. "Other profile keys" expandable list for arbitrary key/value entries the agent has set.
- **Preferences** — session-resume window, default model, embedding model (display only), session title style (auto / manual / timestamp), theme (system / light / dark).
- **Account** — signed-in-as, sign out, last login.
- **Backup & export** — last snapshot time, "Export everything as JSON," "Restore from snapshot" (gated).
- **Danger zone** — clear conversation history, reset profile, drop all measurements, delete database. Each requires typed confirmation.

## 2. Navigation model

### Decision: **bottom tab bar on mobile, persistent left rail on laptop, shared header.**

The user is mobile-first. Thumbs reach the bottom of an iPhone screen comfortably; the top-left is where a stretched hand goes to die. Bottom tabs are the iOS-native idiom and what James already pattern-matches when he picks up his phone.

On a laptop, a bottom tab bar wastes vertical space and looks toylike. A 240px left rail is the SaaS-native idiom and gives room for breathing labels + secondary context (current session title, sync status).

Both layouts share a slim **top header** for: session title, "new chat" affordance, overflow menu (where Settings link lives on mobile if it's not in the bottom bar).

### Mobile (≤768px)

```
┌─────────────────────────────────┐
│  ◀  Session title         ⋯    │  ← top header (44pt)
├─────────────────────────────────┤
│                                 │
│                                 │
│        active view              │
│        (Chat by default)        │
│                                 │
│                                 │
├─────────────────────────────────┤
│  💬     📥    🧠    ✓    ⚙    │  ← bottom tab bar (5 slots)
│ Chat  Inbox Memory Tasks Set    │
└─────────────────────────────────┘
```

**Bottom tab choices (5 slots):** Chat, Inbox, Memory, Tasks, Settings.

**Sources is the cut from the bottom bar** — accessed from Settings → Sources (it's an admin-flavored view, not a daily-tap destination).

### Laptop (≥768px)

```
┌──────────┬──────────────────────────────────────────────────┐
│ Lifecoach│  Session title                           ⋯  +new │
│          ├──────────────────────────────────────────────────┤
│ 💬 Chat  │                                                  │
│ 📥 Inbox │                                                  │
│ 🧠 Memory│            active view (Chat by default)         │
│ ✓ Tasks  │                                                  │
│ 📥 Source│                                                  │
│ ⚙ Setting│                                                  │
│ ──────── │                                                  │
│ Sync: 2m │  ← always-visible status footer                  │
│ Inbox: 3 │                                                  │
└──────────┴──────────────────────────────────────────────────┘
```

Laptop rail shows all 6 views (Sources gets its slot back) plus a status footer with always-visible signals (see §7).

### Routing

TanStack Router, file-based:

```
src/routes/
├── __root.tsx              # Layout shell (rail on lg, tabs on sm)
├── index.tsx               # Chat (new session)
├── c/$sessionId.tsx        # Chat (specific session)
├── inbox/
│   ├── index.tsx           # Insight list
│   └── $insightId.tsx      # Insight detail (modal route on mobile)
├── memory/
│   ├── index.tsx           # Redirects to /memory/facts
│   ├── facts.tsx
│   ├── documents.tsx
│   ├── measurements.tsx
│   ├── reflections.tsx
│   └── search.tsx
├── tasks/
│   └── index.tsx
├── sources/
│   ├── index.tsx
│   └── $connectorId.tsx    # Connector detail (modal route on mobile)
└── settings/
    ├── index.tsx           # Profile + preferences (collapsible sections)
    └── danger.tsx          # Isolated route for destructive actions
```

Modal routes use TanStack Router's `parseRoute` / search-params pattern so deep-links survive page reload. On mobile they render as bottom sheets; on laptop as right-side panels.

## 3. Component hierarchy (top-level shell)

```
<App>                                    # router + query client
  <Shell>                                # layout chrome, viewport-aware
    <HeaderBar />                        # session title, +new, overflow
    <RailNav viewport="lg" />            # left rail (lg+)
    <Outlet />                           # the matched route view
    <TabBar viewport="sm" />             # bottom tabs (sm)
    <GlobalStatus viewport="lg" />       # rail footer (sync, inbox count, agent state)
    <CommandPalette />                   # Cmd+K on lg, search icon on sm
    <SheetHost />                        # registry-driven sheets/drawers
    <ToastHost />                        # transient notifications
    <ConfirmDialog />                    # blocking confirms (danger zone)
  </Shell>
</App>
```

### Sheets vs full views — the rule

A surface is a **sheet/drawer** (not a route view) if any of these is true:

- It's a focused action (edit one thing, ingest one file, confirm something)
- It's transient (closes after one action and returns you to where you were)
- It's invoked from inside a view as a detail of a list item

A surface is a **route view** if any of these is true:

- It's a destination someone might deep-link to
- It's a place you might dwell for >30 seconds
- It has its own non-trivial information architecture

Concretely:

| Surface                          | Type                | Why                                          |
|----------------------------------|---------------------|----------------------------------------------|
| Edit a fact                      | Sheet               | Focused action, one fact                     |
| Ingest a file                    | Sheet               | Focused action; closes when done             |
| Fact detail (view)               | Sheet               | Detail of a list row                         |
| Document preview                 | Sheet               | Detail of a list row                         |
| Connector OAuth                  | Full-page redirect  | External provider hands us back              |
| Profile edit                     | Inline in Settings  | Form, not a separate surface                 |
| Past-conversation browser        | Sheet (left-slide)  | Invoked from Chat header; transient          |
| Insight detail                   | Sheet               | Detail of a list row                         |
| Slash-command autocomplete       | Inline in composer  | Native prompt-kit behavior                   |
| "What did the agent just do?"    | Toast w/ undo       | Transient                                    |
| Memory search                    | Route (`/memory/search`) | Deep-linkable; dwell time              |

### Component inventory (props + role, not implementations)

- **`<Shell>`** `{ children }` — viewport-aware layout container; chooses rail-or-tabs.
- **`<HeaderBar>`** `{ title, actions?, onBack? }` — top chrome; on mobile shows back when not at root of a tab.
- **`<RailNav>`** `{ items, activeId }` — laptop left rail.
- **`<TabBar>`** `{ items, activeId, badges }` — mobile bottom bar; supports per-item badge for unread/error.
- **`<GlobalStatus>`** `{ }` — reads from a status store; renders rail footer signals.
- **`<CommandPalette>`** `{ open, onOpenChange }` — global Cmd+K palette: jump-to-view, recent sessions, run slash-command.
- **`<SheetHost>`** — declarative sheet registry; route components push sheets by id.
- **`<MessageList>`** — wraps prompt-kit's message renderer; injects `<ToolCallDisclosure>` for tool messages.
- **`<ToolCallDisclosure>`** `{ tool: ToolUse, status }` — inline collapsible showing tool name, input, output, error.
- **`<Composer>`** — wraps prompt-kit `PromptInput`; adds slash-command autocomplete + file-drop overlay.
- **`<InsightCard>`** `{ insight, onDiscuss, onAct, onDismiss, onSnooze }` — Inbox card.
- **`<FactRow>`** `{ fact, onEdit, onForget }` — Memory list row.
- **`<MeasurementChart>`** `{ metric, range }` — time-series sparkline / large chart.
- **`<ConnectorCard>`** `{ connector, onSync, onDisconnect, onViewActivity }` — Sources card.
- **`<DropZone>`** — full-window overlay that activates on dragenter anywhere in the app; hands off to ingest sheet.
- **`<IngestSheet>`** `{ files, onConfirm, onCancel }` — confirm parse/embed/extract options before kicking off ingestion.
- **`<SessionListSheet>`** `{ onSelect, onNew }` — left-slide drawer of past sessions (Chat view).
- **`<ConfirmDialog>`** `{ title, body, requireTyped?, onConfirm }` — destructive confirms.

## 4. Key user flows

### 4.1 Send a chat message that triggers a tool call

The flow most likely to define the feel of the app. The agent is tool-heavy (`recall`, `remember`, `set_profile`, `list_recent_interactions`, and soon `add_task`, `complete_task`, calendar-read, etc.). The UI must make tool use **legible without being noisy**.

**Happy path:**

```
1. User taps composer, types "did I complete the labs follow-up task?"
2. User taps send (or presses Enter on laptop).
3. Composer clears; user message appears immediately as a bubble.
4. Streaming indicator appears under the user message ("thinking…").
5. Agent decides to call `recall` tool.
   → A ToolCallDisclosure appears INLINE in the message stream:
      ┌─ 🔎 recall ──────────────────── running ─┐
      │  query: "labs follow-up task"            │
      └──────────────────────────────────────────┘
6. Tool completes.
   → Disclosure updates to "complete" with a chevron:
      ┌─ 🔎 recall ────────── 3 hits ▸ 0.4s ─────┐
      └──────────────────────────────────────────┘
   (Tap to expand and see the hits.)
7. Agent streams the answer:
   "Yes — you marked it complete yesterday at 4:12pm. The follow-up
    notes are in your blood-work doc from 5/12."
8. Streaming ends. A subtle "actions" row appears under the assistant
   message: 📋 Copy · 🔁 Regenerate · 🔗 Cite (expand sources).
```

**Tool call visual rules:**

- One disclosure per tool call, rendered between user message and assistant message.
- Running state: subtle pulse, tool name + input summary, no full payload.
- Complete state: tool name + result summary (count, duration), collapsed.
- Error state: red accent, error message visible without expanding, "Retry" affordance.
- Tapping expands the full input/output as a code-styled block (monospace).
- Multi-tool sequences render as a stacked list under one heading: "Used 3 tools (recall, get_profile, recall)" — tap to expand each.

**Edge cases:**

- **Long-running tool (>5s):** show elapsed time ticking up; offer cancel button on the disclosure after 10s.
- **Tool error:** assistant continues but acknowledges in text ("I tried to look that up but the recall failed — try again?"). Disclosure shows red.
- **Tool requires confirmation** (future: destructive Todoist write): disclosure renders inline with **Approve / Decline** buttons; agent stream pauses until response.

### 4.2 Drop a file to ingest

The agent's data is only as good as what's been ingested. Make this trivially easy on both phone and laptop.

**Laptop (drag-drop):**

```
1. User drags a PDF from Finder over the browser window.
2. <DropZone> overlay fades in across the whole app (any view).
   Copy: "Drop to ingest. We'll parse, embed, and extract."
3. User releases.
4. <IngestSheet> opens with:
     - Filename, size, detected type
     - Optional title (pre-filled from filename)
     - Source (free text, e.g., "Quest labs 5/12")
     - Extract toggles: [x] facts  [x] measurements
     - "Ingest" / "Cancel"
5. User taps Ingest.
6. Sheet collapses to a toast: "Ingesting recipes.pdf… 1/3 stages: parsing"
7. Toast updates as pipeline progresses: parsing → embedding → extracting.
8. On completion: "Ingested. Extracted 4 facts, 0 measurements.
                  Open in Memory ▸"
9. The file now appears in Sources → Recent activity and Memory → Documents.
```

**Mobile (picker + share sheet):**

- Tapping the paperclip in the composer opens iOS document picker (PDF/MD/CSV/image).
- Alternative: the app registers a share target so PDF apps' share sheet can hand off to Lifecoach. On selection, jumps directly into `<IngestSheet>`.
- Camera: a "Scan document" affordance in the composer for paper labs → OCR pipeline (post-v1).

**Slash command parallel:** `/ingest` in composer opens the same `<IngestSheet>`. Useful when James is mid-conversation about a doc he wants to add.

**Error cases:**

- Unsupported MIME: sheet shows "We don't know how to parse `.docx` yet" with a "Track this" link to the roadmap.
- File too large (>20MB): warn but allow.
- Duplicate hash: "This file looks like one already ingested (`recipes.pdf` on 5/12). Re-ingest anyway?"

### 4.3 View today's insights inbox

The morning ritual. This is the view that makes the agent feel like it's doing something while you sleep.

```
1. User opens the app (PWA on home screen, or browser tab).
2. Lands on Chat by default. Bottom tab bar shows badge: "Inbox (3)".
3. Tap Inbox.
4. Top of view: today's date and a one-line agent greeting:
     "Good morning. 2 insights and 1 reflection since yesterday."
5. Scrolling: stack of InsightCards, newest first, grouped by day.
6. First card example:
     ┌─────────────────────────────────────────────┐
     │ HRV trend                            2h ago │
     │                                             │
     │ Your 7-day HRV average is down 12% from     │
     │ the prior week. Sleep duration is unchanged │
     │ — the dip correlates with three late-night  │
     │ Slack sessions Mon/Wed/Thu.                 │
     │                                             │
     │ Sources: HRV measurements (7d), Calendar    │
     │                                             │
     │ [💬 Discuss]  [✓ Acted on]  [🔕]  [⏰ Snooze]│
     └─────────────────────────────────────────────┘
7. Tap "Discuss" → jumps to a new Chat session pre-seeded with the
   insight as a hidden context block; agent's first message:
     "I noticed your HRV trend. Want to dig into the late-night work
      pattern or look at the data first?"
8. Tap "Acted on" → card marks acted-on (gray, collapsed); badge count drops.
9. Tap dismiss icon → card removed from view with undo toast.
10. Bottom of inbox: "Reflections" section, collapsed. Tap to see today's
    daily reflection + recent weeklies.
```

**Empty state:** "Nothing to surface this morning. Your last reflection was [date, time]. Talk to your coach about anything you want them to think about."

### 4.4 Trigger a Todoist sync manually and see the result

The user is on Tasks, knows they just added something in the Todoist app, and wants it pulled in *now*.

```
1. User is on /tasks. Header right-side action: ↻ Sync (last 12m ago).
2. Tap Sync.
3. Button enters loading state; label changes to "Syncing…"
4. Behind the scenes: connector hits Todoist API, diffs against local task table.
5. On success (~1-3s), button returns to ↻ Sync (just now).
6. A short slide-down banner under the header shows the diff:
     "+3 added · 1 completed · 2 due-date changes"
   Banner persists for 8s or until tapped.
7. The lists below have updated. New tasks have a faint "new" pip
   for 60 seconds. Completed tasks animate out with strikethrough.
8. Tap the banner → opens a sync-detail sheet listing every change
   with a per-change "undo on Todoist" affordance (future).
```

**Error case:** "Sync failed — Todoist rate limit. Retry in 45s." Banner is red; button shows ⚠ and the cooldown.

**Auto-sync:** scheduled sync runs every 15 minutes in the background. The "last synced" timestamp updates everywhere it appears (Tasks header, Sources card, rail footer on laptop). No banner shown unless deltas are notable (>5 changes or first-of-day).

### 4.5 Edit a profile fact

The agent stores arbitrary facts via the `remember` tool. The user occasionally needs to fix one (e.g., "I'm not vegetarian anymore").

**Path A: from Memory → Facts.**

```
1. /memory/facts. Search "vegetarian" or filter by category "preference".
2. Tap the row.
3. <FactDetailSheet> slides up from bottom (mobile) / in from right (laptop):
     ┌─────────────────────────────────────────┐
     │ Vegetarian since 2019      [⋯]    [×]  │
     ├─────────────────────────────────────────┤
     │ Category: preference                    │
     │ Subject:  diet                          │
     │ Body:     "Vegetarian since 2019…"      │
     │ Confidence: ████████░░  0.85            │
     │ Valid from: 2019-03-01                  │
     │ Valid to:   (open)                      │
     │ Source: chat, 2026-04-12                │
     │                                         │
     │ [Edit]  [Set valid-to today]  [Forget]  │
     └─────────────────────────────────────────┘
4. Tap Edit → fields become editable inline. Save commits to memory.
5. "Set valid-to today" is a shortcut: marks the fact superseded as of
   now without deleting it (preserves episodic truth).
6. Forget → confirm dialog ("Forget this fact? The agent won't recall
   it but the message that created it stays in episodic history.").
```

**Path B: from chat (more common).**

The user types: "I'm not vegetarian anymore — eating fish since last month."
The agent calls `remember` with a new fact (or sets `validTo` on the old one). The user sees a tool-call disclosure for the write. **Critical:** the assistant's textual reply should explicitly say what it changed ("Got it — I marked the vegetarian fact as ending in April and saved a new pescatarian preference"), and that line should have a "View" link that opens the new fact's detail sheet. This is the trust loop.

### 4.6 Browse and search past conversations

Episodic memory is rich (every message persisted). The user needs to find a thread from two weeks ago: "what did I tell my coach about my dad's surgery?"

**Anatomy:** invoked from Chat header — a left-slide sheet (`<SessionListSheet>`).

```
1. On Chat, tap the session title in the header (or the hamburger on mobile).
2. <SessionListSheet> slides in from the left.
3. Top: search input ("Search your conversations…") + filter (date range).
4. Below: chronological list of sessions, newest first, each row:
     - Auto-summary (one line, generated by agent at session end)
     - Started-at relative time
     - Message count badge
5. Typing in search runs semantic recall over the messages scope.
   Results show as message snippets with the session as context,
   sorted by relevance:
     ┌──────────────────────────────────────────┐
     │ "…dad's surgery was on the 14th, recovery │
     │  has been slower than expected…"          │
     │ Wed 5/14, 9:02pm · in 'Tuesday check-in'  │
     └──────────────────────────────────────────┘
6. Tap a result → opens that session in read-only mode with the
   matched message scrolled into view + highlighted.
7. From a read-only historical session, a "Continue this thread"
   button at the bottom starts a NEW session pre-seeded with the
   prior session's summary as context (does not mutate history).
```

**Empty state:** "No conversations match." with a "Reset search" link.

## 5. Mobile considerations

The dominant device. Treat laptop as the bonus.

### What changes on mobile

| Element                       | Mobile (≤768)                         | Laptop (≥768)                       |
|-------------------------------|---------------------------------------|-------------------------------------|
| Primary nav                   | Bottom tab bar                        | Left rail (240px)                   |
| Session title in Chat         | Centered in header, tap to open list  | Inline in header + rail context     |
| Past-session browser          | Bottom sheet, full-height             | Left-side panel, push or overlay    |
| Insight detail                | Bottom sheet                          | Right-side panel                    |
| Fact / document detail        | Bottom sheet                          | Right-side panel                    |
| Command palette               | Search icon → modal, full-screen      | Cmd+K → centered modal              |
| Settings layout               | One section per screen, drill-down    | All sections visible, anchored      |
| Memory tabs                   | Sticky horizontal scrollable tabs     | Tabs visible inline                 |
| Ingest sheet                  | Bottom sheet, swipe to dismiss        | Centered modal                      |
| Confirm dialogs               | Bottom-anchored action sheet          | Centered modal                      |
| Status footer (sync/etc.)     | Hidden — accessible from header overflow | Persistent in rail footer        |
| Sources view                  | Tucked under Settings (one less tab)  | Top-level rail item                 |

### Touch targets

- Minimum 44pt for every tappable affordance. prompt-kit defaults satisfy this; verify per shadcn component.
- Action icons in cards (the row of Discuss/Act/Dismiss on insight cards) use 44pt touch zones even with smaller icons.
- Avoid double-tap or long-press as the *only* affordance for any action. Long-press allowed as an accelerator (e.g., long-press a fact row to open a quick-actions menu).

### Gestures

- **Swipe left on a row** (insight, fact, task) → quick-action drawer with primary destructive action (Dismiss / Forget / Complete). iOS-native pattern.
- **Swipe right on a row** → secondary action (Snooze / Edit). Optional, restraint preferred.
- **Pull to refresh** on Inbox, Tasks, Sources — triggers a manual sync of the relevant data.
- **Swipe down on a sheet** → dismiss (prompt-kit / shadcn sheet defaults).
- **Edge-swipe back** (iOS) → router back. Standard.
- **No swipe-between-tabs.** Tabs are destinations, not pages of one document. Swiping between would confuse with horizontal scroll patterns inside views.

### One-handed reachability

- Composer pinned to bottom (above tab bar) is in the thumb sweet spot.
- Header back affordance is also at the top-left (out of thumb range) — that's fine because mobile users get a swipe-back gesture for free.
- Anything destructive lives in overflow menus (top-right) or requires a sheet with explicit confirmation — never one-tap-reachable.

### PWA & install

- Manifest configured for "add to home screen" — Lifecoach should feel like an app, not a website. No browser chrome when launched from home screen.
- Status bar styled to match app header (dark or matching theme).
- Splash screen with the wordmark, calm.
- Offline state: not supported in v1 (Tailscale-only), but show a graceful "Reconnecting…" banner when the connection drops.

### Notifications

Out of scope for v1 (see §8). When added: PWA push notifications for high-priority insights only, with an opt-in per category.

## 6. Component-level conventions (cross-cutting)

A handful of patterns reused everywhere; documenting once.

### Empty states

Every list view has a designed empty state that:
- Names what would be here ("No facts yet.")
- Explains how things get here ("Facts appear as you chat or ingest documents.")
- Offers an action ("Start a conversation" or "Ingest a file")

Never show a bare "No results" with no next step.

### Loading states

- **Initial view load:** skeleton rows matching the eventual layout. No spinners on initial loads.
- **Action in progress** (sync, send, ingest): inline progress on the originating affordance, never a full-screen blocker.
- **Streaming chat:** prompt-kit's native streaming indicator under the user message.
- **Background work** (auto-sync, reflection cron): not surfaced unless the user is in a relevant view; result surfaces via banner/badge.

### Error states

- **Recoverable** (network blip, rate limit): inline banner near the source, with retry. Disappears on success.
- **Validation** (form error in settings, ingest config): per-field message under the field.
- **Catastrophic** (DB unreachable, agent core not responding): full-view boundary with a "Lifecoach can't reach your memory right now" message and a retry. Don't blame the user; this is single-user — it's the system's problem.

### Confirmation patterns

- **Reversible action:** no confirmation; show an undo toast for 8s. Examples: dismiss insight, mark task complete, end session.
- **Destructive but undoable:** single-tap confirm dialog. Example: forget a fact.
- **Destructive and irreversible:** typed-confirm dialog ("Type 'forget all measurements' to continue"). Example: anything in Settings → Danger zone.

### Time formatting

- **Within last hour:** "12m ago"
- **Today:** "3:14pm"
- **Yesterday:** "Yesterday, 9:02pm"
- **This week:** "Wed, 9:02pm"
- **Older:** "May 14, 2026"

All timestamps have a tooltip / long-press with the absolute time.

## 10. Cross-flow behaviors worth calling out

A few behaviors that span multiple views and should be designed consistently.

### "Discuss in chat" pattern

Used from: Inbox (insight), Memory (fact, document, measurement), Tasks (task), Sources (ingestion event).

Behavior: starts a **new session** with a hidden context block referencing the source object (insight id, fact id, etc.). The agent's first message references the context. The user can see the source by tapping a chip under the agent's first message ("Context: Insight 'HRV trend' · view ▸").

This is the connective tissue that makes the rest of the app feel like a coach, not a CRUD UI. Every non-chat view should have a way to drop the user into Chat with context.

### "What did the agent just do?" pattern

After any tool call that mutates state (set_profile, remember, complete_task, add_task), the Chat view shows the tool-call disclosure and the assistant message explicitly names what changed. On other views, if data the user is looking at is mutated by an agent action, the affected row briefly highlights when next rendered.

### Optimistic updates

- Sending a message: user bubble appears immediately, streaming state begins.
- Editing a fact: change persists locally, syncs in background; revert with toast on error.
- Dismissing an insight: card animates out immediately, undo toast.
- Marking a task complete: row strikes through immediately; reverts on sync failure.

### Search behavior

There are three search surfaces:
- **Memory → Search** (`/memory/search`): semantic recall, all scopes, faceted by ref type.
- **Sessions search** (inside Past-sessions sheet): scoped to messages.
- **Command palette** (Cmd+K): jump-to-view, recent sessions, slash-commands.

Same underlying recall infra. Distinct UX surfaces because they serve distinct intents.

## 11. Friction reduction opportunities (gap analysis)

Things noticed while writing this spec that warrant follow-up either in design or roadmap.

1. **First-launch experience.** The roadmap assumes the agent already knows James. But the *web* UI is new — first launch on a phone is a moment. Suggestion: a one-screen welcome ("You're signed in. Last chat: [date]. Catch up?") that's not a tour, just orientation.
2. **The "trust" signal for agent writes is verbal-only.** The agent says "I remembered X." We could strengthen this with a small persistent affordance — a sidebar/sheet that shows "Things saved this session" — so the user can audit before they end a session. Worth prototyping.
3. **No way to mark a fact as "wrong" vs "outdated."** Editing is binary. A "this was never true" affordance (vs "this was true but isn't anymore") would help the agent learn its own error modes. Maps to a `superseded_reason` field on facts.
4. **Reflections are passive.** They appear in Memory and Inbox. There's no "ask the coach to reflect on X right now" entry point. Slash command `/reflect [topic]` would close the loop.
5. **Tool-call disclosures could overwhelm a long answer.** If the agent calls 8 tools to answer one question, the message stream becomes a wall. Suggestion: collapse all tools used for one assistant message into a single disclosure: "Used 8 tools (4 recall, 2 get_profile, 2 remember) ▸". Already noted in §4.1 — but worth flagging as a real risk once integrations expand.
6. **Insight cards have no thread continuity.** If the user discusses an insight in chat, the insight itself doesn't know about that conversation. A "Discussed [date]" chip on the card would help.
7. **Sources view assumes file-drop is local.** Once the user is on a phone (no file system access in the same way), "file drop" needs reframing as "uploads from this device." Worth a separate copy pass.
8. **Tasks view risks being a worse Todoist.** Keep it small: today + week + agent activity. If we add a kanban or list filters, we've lost the plot.
9. **Settings → Danger zone needs offline backup verification.** Before any "delete" action, show "Last backup: [date]. Backup is older than 24h" warning if true.

## 13. Open questions for the visual designer / frontend architect

Not for me to answer, but worth raising for the next handoff:

1. **Dark mode default.** Health data on a black background reads "calmer." Recommend dark as the system default, but follow OS preference.
2. **Voice tone of agent.** Copy decisions (greeting, insight phrasing, error messages) need a voice-and-tone pass — beyond this spec.
3. **Avatar / agent identity.** Is there an avatar for the agent in chat bubbles? Recommend: no human face. A subtle mark or color block. (prompt-kit supports both; default to none.)
4. **Cross-device session continuity.** When James starts a session on his phone and opens the laptop, should the laptop resume the same session or start fresh? Recommend: resume with a "you were last here on iPhone" banner.
5. **PWA install prompt timing.** When do we surface "add to home screen"? Recommend: after the 3rd visit on iOS Safari only, dismissible permanently.
6. **Haptic feedback on mobile.** iOS supports haptics on PWAs via the Vibration API in limited ways; tasteful use on confirmations could reinforce the trust mood.
